Introduction

As part of most University degrees, students complete a final year dissertation or project. Within scientific degrees, these projects are usually chosen from a list, determined by the academic supervisors. The task of allocating and matching supervisors to projects, and projects to students, can be complicated, and allocation can depend on a number of factors, such as supervisor/student preference and project availability.

Hence why it makes sense for this to be done through a unified online system, where students log in to rank project preference, module leaders log in to allocate students and supervisors to projects, and supervisors view their allocation.

Given this brief, a strict style guide, and a full-stack framework, I set out to create such a system for a client (my lecturer).

The specification

A specific brief was given for this task:

'The .. dissertation module leaders .. need a web based system that allows students to chose projects within particular themes, and to browse possible project topics in those themes. They assign a rank ordering to the themes they pick (first choice, second choice...). Supervisors and theme leaders must be able to upload information about possible topics within their themes and provide a general description for the theme as a whole. Topics could possibly be available across several themes and may be marked as not being offered (and thus hidden from students). The module leaders need to be able to see lists of sttudents and the themes they have chosen, possibly in a variety of orderings and filterings. Once selectiosn have been arbitrated by the module leaders, theme leaders need to be able to see the list of students that have selected their theme and then allocate them, to individual supervisors - the system should automate the sending any relevant email messages about the allocations.'

The Stack

The back-end language of the framework was PHP. This interfaced with a MySQL database through the Redbean ORM, and rendered pages for the front-end using Twig. This was deployed to an Apache webserver. The front-end stack consisted of HTML5, Bootstrap and JQuery.

Project Planning

So that the 'client' had some input and idea of what the finished product would look like, a fontboard, moodboard, and storyboard of the website were created.

This storyboard gave some inspiration about the color scheme and 'feel' the website would have. From this a fontboard could be created. Unfortunately given the breif comic sans was off the cards, so in the end Roboto was used as the font of choice.

It's worth noting that the font wasn't selected until after the storyboard was presented, so Arial is the actual font used on the storyboard.

Take-homes

The full code and PHPDoc for this project can be found on my Github, so I will not be discussing implementation details in this blog. But in this section I will be talking about some of the key lessons learned.

ORM is OResome

As this was the first time I had used Redbean, and first time I had used in an actual project, I found the learning curve steep, particularly when it came to handling joins. However once I got to grips with it I saw it's benefits, and abstracting away from raw SQL made perfect sense.

Clearly planning pays dividends...

This was also the first project where I'd planned in such detail. I found that having a clear idea of what screens should look like, and how users should flow through the app saved time and confusion during development. I never got confused, creating a screen that wasn't necessary and having to backtrack, as the storyboard was always there to keep me on track. In all my subsequent projects I have used this storyboarding approach, even if in a more diluted or rough form.

...But sometimes you need to be realistic about what's feasible

However, during development it became clear that what I was attempting to create just wasn't feasible in the time scale I was working with. Some aspects of usability and design had to be compromised, so that the deadline was met. The lesson here being it's important to evaluate how long something will take, and how much value it will actually add to the project, in order to ensure time is there for the most important aspects of the project.

Not unit testing will lead to regrets later on

Whilst I have doubts about whether I could have delivered on an extensive set of unit tests, having some would have been desirable in the latter stages of development. As my codebase grew and became more complex, isolating errors became increasingly more difficult. When it got to the last stage of the project, and there was real time pressure, debugging the error that breaks everything would have been alot less stressfull if a set of unit tests gave clues about where and what the error was.

In A later blog post I'm going to talk about another project, where I provided extensive unit tests, and I found that even though the codebase in this project was far larger and more complex, debugging was still less stressful as a result.

The finished Product

I was pretty happy with the system I built, and so was the 'client'. I delivered on all of the basic requirements, and couldn't spot any errors (though I don't doubt there probably were some). As can be seen from some of the screenshots below the final product does differ from the storyboard. This is due to a combination of the fact that bootstrap has certain constraints in the way things look on a page, and I discovered limitations in what was possible as mentioned earlier.
